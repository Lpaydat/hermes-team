Decision recorded

Cites: inputs weighed, alternatives compared, winner, rationale, supersession pointer
📜 ADR-0XX: [Decision Title]
Status: Accepted  |  Date: 2026-07-11
Context: Why this decision was needed — the problem, constraints, inputs
Decision: What we chose and why — the winner from the alternatives
Alternatives weighed: Option A (rejected because…), Option B (rejected because…)
Consequences: What this enables, what it costs, what it locks in
Supersedes: ADR-0YY (if applicable) — history is never rewritten

⚠️ Changing an ADR requires an architecture ticket — never a dev-loop card.

T2 — Design-it-twice
Fan out 3 independent architecture candidates
3 radically different stacks compared
A: Django monolith + Celery + PostgreSQL
B: FastAPI + Redis + MongoDB
C: Next.js + tRPC + PostgreSQL + Drizzle
Winner: C — chosen for type safety end-to-end, smallest team footprint, best DX velocity. Rejected A: monolith too heavy for the scraping workload. Rejected B: MongoDB adds migration risk.
codebase-design domain-modeling
📜 ADR-001: Use Next.js + tRPC + PostgreSQL + Drizzle ORM
Status: Accepted  |  Tier: T2
Context: Recipe cost SaaS, solo dev, need type safety + scraping + low ops overhead
Decision: Next.js (fullstack) + tRPC (type-safe API) + PostgreSQL (relational) + Drizzle (typed ORM)
Alternatives: Django monolith (rejected: heavier ops), FastAPI+Mongo (rejected: migration risk)
Consequences: TypeScript everywhere, serverless scraping needs external worker, Prisma rejected for query control
→ tech-lead slices ADR-001 into tickets → developer implements
Architect role: answer questions, verify ADR adherence, gate any new changes
