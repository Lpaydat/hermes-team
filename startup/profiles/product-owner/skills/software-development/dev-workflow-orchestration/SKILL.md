# Dev Workflow Orchestration

You are the single front door for development work. You own WHAT gets built and in what priority order. Tech-lead owns HOW it gets built. This separation is the core invariant — violating it is the most common failure mode.

## The WHAT vs HOW boundary

| WHAT (yours) | HOW (tech-lead's) |
|---|---|
| Grill the user (`grilling`) | Technical design / architecture |
| Write PRD (`to-prd`) | Contract / evals / acceptance criteria refinement |
| Decompose into slices (`to-issues`) | Decompose a bead into dev/verifier cards |
| Create beads + dependencies | Harness delegation (pi/zz/claude) |
| Prioritize which bead is next | Iterate on verification failures |
| Observe loops, steer if needed | Execute the failure-fix cycle |

**Never cross into HOW.** Don't write function signatures, don't specify edge cases, don't create dev/verifier cards, don't invoke the harness, don't manually unblock tasks mid-loop. If the loop is broken, STOP, take notes, and notify the user — don't fix it by interfering.

> **Naming note (Jul 2026)**: The `reviewer` profile was renamed to `verifier` based on academic research — see [references/generator-verifier-research.md](references/generator-verifier-research.md). The profile does both dynamic verification (runs tests) AND static review (adversarial diff via `/review` skill). "Verifier" is broader and matches the generator-verifier academic consensus.

## The full flow

### Phase 1: In-session planning (you + user)

```
User brings idea/bug/request
  ↓
Grill the user (grilling skill) — one question at a time, recommended answer attached
  ↓
Write PRD (to-prd skill) — what to build, user stories, acceptance criteria, success metrics
  ↓
Write ADRs (decision-mapping + domain-modeling) if architecture decisions are needed
  ↓
Decompose into slices (to-issues skill) — tracer-bullet vertical slices, NOT horizontal layers
  ↓
Quiz the user on the breakdown — granularity, dependencies, merge/split
  ↓
Create ALL beads + dependencies (bd issue create with --parent, --deps)
  ↓
Session ends → beads persist as the masterplan → you forget the conversation details
```

**Critical**: Create all beads in ONE session while you have full PRD context. Do not create beads one at a time across sessions — you'll lose the dependency graph and the prioritization reasoning. Beads ARE the persistent memory of the plan.

### Phase 2: Automation loop (cron, no user)

**Dependency-driven dispatch.** Multiple independent beads can be dispatched simultaneously, up to the per-profile concurrency cap. Bead dependencies control ordering (what CAN start); the cap controls resource safety (how many run NOW). See [references/dependency-driven-parallelism.md](references/dependency-driven-parallelism.md) for the full parallelism design.

```
Script runs bd ready on all active projects
  ↓ (nothing ready → exit silently, zero token cost)
Ready beads exist (one or many)
  ↓
For each ready bead:
  Check running count for tech-lead against cap (max_in_progress_per_profile)
  ↓ (at cap → skip, next cron tick will retry)
  Create kanban card for tech-lead (idempotency-key dedup)
  ↓
Tech-lead receives card → runs loops-engineering (HOW)
  ↓
Tech-lead creates dev/verifier cards → loop executes autonomously
  ↓
Loop completes → kanban card archived
  ↓
Next cron tick → check bd ready again
```

**The mechanical part stays free.** The workflow engine (`scripts/workflow-engine.py`) is a single zero-token cron (1min) that runs three phases in order: (1) bead-sync (sync card→bead status, close done beads), (2) auto-dispatch (bd ready → create tech-lead cards with `--idempotency-key "bead-<id>"` dedup, `--workspace "worktree:<repo-path>"` for parallel isolation), (3) board-scanner (detect blocked tasks → escalate to proper profile). The old three separate crons (auto-dispatch.sh, bead-sync.py, board-scanner.py) were combined into one script for ordering guarantees and simpler management. The dispatcher's `max_in_progress_per_profile` config is the real concurrency cap. Deploy as `no_agent=true` cron (every 1 min). Zero tokens unless there's work.

**Parallel execution** (PROVEN Test 22 + cleanroom + C6 3/3, Jul 2026): with `max_in_progress_per_profile: 3`, independent beads run through the full pipeline simultaneously. Merge-slot serialization (`bd merge-slot`) ensures only one verifier merges to main at a time. **Merge conflict resolution PROVEN 3/3**: when two parallel beads modify the same file, the verifier detects the conflict on rebase, correctly FAILs ("conflict resolution is code-writing, I never write code"), creates a fix card for the developer who resolves it, new verifier re-reviews and PASSes, tech-lead re-blocks on the new verifier via `delegate_and_wait`, then completes after final merge. See [references/dependency-driven-parallelism.md](references/dependency-driven-parallelism.md) for the full design, test proof, and conflict handling protocol.

## Beads vs Kanban — the two-layer model

| Layer | What it is | Lifetime | Who writes |
|---|---|---|---|
| **Beads** | Masterplan — epics, beads, dependencies, acceptance criteria | Durable (survives across sprints) | PO creates (`bd create`), verifier closes on merge (`bd close`) |
| **Kanban** | Execution — one card per bead being worked on RIGHT NOW | Ephemeral (archive after done) | PO creates tech-lead card, tech-lead creates dev/verifier cards |

Beads answer "what needs building and in what order?" Kanban answers "who is working on what right now?"

**These two systems don't conflict — they track different things at different layers.** A bead can be `open` while multiple kanban cards related to it are `done` (each completed their piece). The bead only closes when the verifier merges. `bd close` is the bridge: it closes the bead AND completes the verifier card simultaneously, then triggers `bd ready` to promote dependent beads → auto-dispatch creates the next card.

**Who touches bd:**
- PO: creates beads, creates merge-slot, reads (`bd ready`, `bd show`)
- Verifier: acquires/releases merge-slot, closes beads on merge
- Tech-lead: reads bead ACs (`bd show`), closes beads (`bd close`)
- Developer: reads bead ACs (`bd show`) — read-only

**Idempotency keys are stored in SQLite but NOT exposed in the JSON API.** `kanban create --idempotency-key "X"` stores the key in the `tasks.idempotency_key` column (with an index), and dedup works at CREATE time (returns existing card instead of creating a duplicate). But `kanban list --json` omits the field. To inspect: `sqlite3 kanban.db "SELECT id, idempotency_key FROM tasks WHERE idempotency_key IS NOT NULL"`.

## `/review` in the verifier process

The `/review` skill (mattpocock) is a **tool** the verifier uses for static analysis — not a role name. It checks git diffs along Standards + Spec axes. The verifier integrates it as Phase C of its two-phase protocol (static analysis, parallel with the fresh-eyes subagent).

**Reference layers for `/review`**: contract.md (tech-lead's technical spec) + bead acceptance criteria. NOT the PRD — it's too high-level for per-slice review. If the verifier finds a spec gap (code meets contract but contract diverges from PRD intent), that escalates to PO.

### The two-phase verification protocol (anti-confirmation-bias)

The verifier runs BOTH a delta check AND a fresh-eyes pass on EVERY iteration — not just checking whether prior findings were fixed. This prevents confirmation bias (backward-only verification degrades over iterations, per Huang et al. "LLMs Cannot Self-Correct Reasoning Yet").

**Phase A** (delta, iterations 2+): re-run prior findings' repros — FIXED / STILL-FAILING / REGRESSED.

**Phase B** (fresh-eyes, every iteration): `delegate_task` subagent with deliberately restricted context — gets ONLY contract + bead ACs + diff + worktree path. NEVER prior findings, developer report, or trace. This is the independence guarantee.

**Phase C** (static analysis, parallel with B): `/review` + `ponytail-review` on the diff.

Then synthesis → gap probing → verify-findings gate → AC checklist gate → verdict.

**Key routing decision**: on FAIL (code bug), the verifier creates a fix card assigned to **developer directly** (warm-resume harness with findings) — NOT tech-lead. Tech-lead is only involved at iteration ≥ 3 (circuit breaker) or spec gaps (contract is wrong). This keeps the loop tight: developer fixes → verifier re-verifies, no planner overhead per iteration.

**Verifier uses `delegate_task`, not harness**: the verifier is judging code (read + execute), not writing it. The harness (pi/zz) is a code-writing tool for the developer. Different tools for different jobs.

### The AC checklist gate (step 7 — prover-verifier pattern)

The verifier's most important gate (designed Jul 2026). The bead acceptance criteria (PO-written `- [ ]` checkbox items) are the **product contract**. Both developer and verifier must engage with each AC:

| Step | Who | What |
|------|-----|------|
| **Write ACs** | PO (in bead) | Checkbox items — what the slice must deliver |
| **Provide proof** | Developer (in completion report) | Maps each AC to a test + output — a **claim** |
| **Re-verify each** | Verifier (step 5) | Writes own probe, executes independently — the **fact** |

If developer claims "AC met" but verifier's independent probe fails → **Critical finding**. This catches test-tampering and tests that test the wrong thing.

## Infrastructure gaps identified (not yet resolved)

### Sandboxing
No filesystem isolation exists today — every profile runs as the same OS user. Developer's pi harness can write to ANY directory, verifier can modify code during verification, and a bug in generated code could delete files outside the workspace. For enterprise use, this needs either container-per-task (Docker/Podman), lightweight sandboxing (firejail/bubblewrap on the pi/zz subprocess), or at minimum separate worktrees with strict permissions. This is an infrastructure concern, not a workflow design issue — the architecture is sound, the execution environment needs hardening.

### Long-running loops (>12-20hr)
Not yet tested. Concerns: context window accumulation across many cron cycles, trace ledger growth, board pollution from archived cards, Z.AI session token expiry, and whether the auto-dispatch cron can sustain indefinite operation without drift. The cron script is zero-token and stateless, so it should be stable — but this needs validation.

### Dependency chain deadlock (discovered Test 20, Jul 2026)
- **Dependency chain deadlock (discovered Test 20, Jul 2026)**: when a parent task blocks (`needs_input` or crash exhaustion), its children stay in `todo` **forever** — nobody is notified, the board freezes. The dispatcher only dispatches by `assignee + status`; it doesn't read comments or traverse parent chains. This is the **single biggest reliability gap** in the autonomous system.

**Root cause**: `blocked` ≠ `cancelled`, but children can't promote while their parent is blocked. There's no cascade-block, no notification, no timeout.

**The self-healing fix**: board scanner — **REDESIGNED + DEPLOYED + PROVEN Jul 2026**. New model: ANY blocked task → escalate to proper profile for LLM inspection. NO auto-unblock (dispatcher circuit breaker handles crashes natively). Scanner is the **unblock bridge**: agents CANNOT unblock foreign tasks (tool API `_enforce_worker_task_ownership` rejects), but scanner CLI can. Proven in production during C6 R1 + 5/5 validation test. See [references/board-scanner-redesign.md](references/board-scanner-redesign.md) for the correct design + ownership source-code analysis + live test proof.

### 429 rate-limit resilience (PARTIALLY IMPLEMENTED Jul 2026)
Previously a pending recommendation — layers 1-2 deployed, layer 3 PAUSED:
1. **In-session retry**: `api_max_retries: 10` (was 3) — agent survives ~5 min of 429s without dying
2. **Kanban reclaim**: existing auto-reclaim for genuine crashes
3. **Board scanner v2**: DEPLOYED + PROVEN — dispatcher circuit breaker (`consecutive_failures >= failure_limit`) handles crash recovery natively. Scanner escalates blocked tasks to proper profile.
4. **Incident logging**: every deadlock/escalation logged to `INCIDENTS.md` for analysis (will resume when scanner is rewritten)
